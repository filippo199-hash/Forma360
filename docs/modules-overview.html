Templates & Inspections

Module 01

Templates define the structure, questions, and logic of every check your team performs. An inspection is a live instance of a template — the actual data capture. A completed inspection becomes a report that can be exported and shared.

Building a Template Template Editor

Every template starts with a Title Page and at least one Inspection Page. On the title page, auto-populating response types capture metadata at the start of every inspection:

  • Site — links the inspection to a location or project.
  • Conducted by — auto-fills with the user who starts the inspection.
  • Inspection date — auto-fills with the start date/time.
  • Document number — auto-increments per template (supports custom alphanumeric schemes, e.g. "AUDIT000001").
  • Location — auto-captures device GPS coordinates and address.
  • Asset — links the inspection to a specific asset.
  • Company — links to a contractor company.

On the inspection pages, the creator adds questions organised into sections. An inspection title format auto-generates unique names from title-page responses (e.g. "[Site] – [Date] – [Doc Number]").

Response Types Question Setup

Each question is assigned a response type:

  • Multiple choice — predefined answers. Each response can have a colour and flagged status. Multi-select can be enabled. Users can create Custom Response Sets with tailored labels and colours, and Global Response Sets reusable across templates.
  • Text — free-text answer.
  • Number — numeric input within a configurable range.
  • Date & Time — date or time picker.
  • Media — photo, video, or PDF upload (up to 10 MB/image, 3 min/video, 20 MB/PDF).
  • Annotation — draw markups on an uploaded image (e.g. floor plan).
  • Signature — digital signature capture (see Multi-Signature below).
  • Slider — visual scale input.
  • Checkbox — simple on/off check.
  • Instruction — read-only text/image/PDF to guide the inspector (not an input).

For every question, the user can optionally add a Note, attach Media, or create/link an Action.

Multi-Signature Flow Signatures ✦ Improved

When adding a signature field to a template, the template creator can now configure how many signers are required. The options are:

  • Single signer — one person signs (current behaviour).
  • Multiple signers — the creator specifies the number of required signers (e.g. 2, 3, 5) and defines a signing sequence:
    • Sequential signing — signers must sign in a defined order. Signer 1 receives the request first; once they sign, Signer 2 is notified, and so on. The inspection cannot be completed until all signers in the sequence have signed.
    • Parallel signing — all signers are notified at once and can sign in any order. The inspection is blocked until all required signatures are collected.
  • For each signer slot, the creator can either pre-assign a specific user or role (e.g. "Site Manager", "Quality Lead") or leave it for the inspector to select at the time of inspection.
  • Each signer's name, timestamp, and signature image are captured individually and appear in the inspection report.

This is critical for workflows like work permits, order sign-offs, safety authorisations, and any form requiring multi-party approval.

Flagging Responses Data Quality

Responses can be individually flagged. When a flagged response is selected during an inspection, it is highlighted in reports and surfaces in Analytics dashboards as an item needing attention. Flagging is available only on multiple-choice response types. Each response option in a custom response set can be independently marked as "flagged" during template design.

Logic Fields Conditional Logic

Logic fields make templates dynamic. Set a condition (e.g. "if the answer is 'No'") and choose triggers:

  • Ask follow-up questions — additional questions/sections appear. Up to 40 nested levels.
  • Require action — the inspection blocks until an action is created or linked for that question.
  • Require evidence — user must attach a photo or video.
  • Require note — user must type an explanation.
  • Send notification — auto-notify specific users, groups, or site members (up to 500 recipients). Fire immediately or on completion.

Multiple triggers can be stacked. Different logic paths can be set for different responses on the same question.

Table Types Structured Data ✦ Improved

When adding a table to a template, the user now selects from pre-built table types:

  • Blank table — a standard empty table. The user defines columns (up to 20) and response types per column. Inspectors add rows (up to 100) during the inspection. Best for general-purpose repetitive data capture.
  • Risk assessment table — pre-structured with columns for Hazard, Risk Description, Likelihood, Severity, Risk Rating (auto-calculated from Likelihood × Severity), and Control Measures. Colour-coded risk levels are applied automatically. Ideal for risk assessments, HIRA, and JSA workflows.
  • Checklist table — pre-structured with Item, Status (Pass/Fail/N/A), and Comments columns. Best for equipment checklists, pre-start checks, and compliance verification.
  • Inventory table — pre-structured with Item, Quantity, Condition, and Notes columns. For stock counts, PPE checks, and equipment inventories.

All table types are fully customisable after selection — columns can be added, renamed, reordered, or removed. Within columns, flagged responses and notifications can be triggered. Tables export cleanly into PDF and Word reports.

Approval Pages Workflows

A template can include an Approval Page. When enabled, the inspector submits for review. The approver reviews and either approves or rejects (with comments). Rejected inspections return to "In Progress". Once approved, it cannot be marked as incomplete. Combined with Multi-Signature, this enables complex sign-off chains.

Template Creation Methods Getting Started

  • From scratch — drag-and-drop editor.
  • AI-generated — describe the purpose in a prompt, upload a photo/PDF, and AI generates the template.
  • Convert from file — import an existing Excel, PDF, or image.
  • Public Library — browse and download from 100,000+ ready-made templates by industry and use case. All fully customisable.

Conducting an Inspection Workflow

The user selects a template (from the list, a schedule, or from within an asset/issue) and works through each page:

  • Offline support — start and complete without internet. Auto-saves locally, syncs on reconnect.
  • Required questions — marked with an asterisk; blocks completion until answered.
  • In-inspection actions — at any question, create or link an action. If logic requires it, the inspection blocks until done.
  • Multi-signature collection — if the template requires multiple signatures, the inspector initiates the signing sequence. The inspection remains open until all signatures are collected.
  • Edit after completion — reopen for editing (back to "In Progress"), unless an approval page was approved.
  • Completion summary — shows all flagged items and pending actions. Bulk-assign or skip.

Scheduling Inspections Automation ✦ Improved

Administrators create inspection schedules assigning templates to users, groups, or sites on a recurring basis. The improved scheduling system includes:

  • Calendar views — schedules are visualised in a daily, weekly, and monthly calendar view (not just a flat list). Colour-coded by template or site for clarity. Users can toggle between views.
  • Drag-and-drop rescheduling — move or extend scheduled inspections by dragging them within the calendar.
  • Frequency options — daily, weekly, fortnightly, monthly, quarterly, annually, or custom intervals.
  • Assignment options — assign to specific users, groups, or sites. When assigned to a group/site, all current members receive the schedule.
  • Automated reminders — configurable reminder notifications before the inspection is due (e.g. 1 day before, 2 hours before).
  • Overdue tracking — missed/late inspections are flagged in the calendar view with clear visual indicators. Managers can see overdue counts per site or user.
  • Completion dashboard — a real-time view showing completion rates by user, site, and time period. Identify who is falling behind at a glance.
  • Workload balancing — when assigning, the system highlights users who already have a heavy schedule, helping managers distribute inspections evenly.

Action Deduplication in Recurring Inspections Workflow ✦ Improved

A common problem with daily or recurring inspections: a user flags an issue on a question and creates an action. The next day, conducting the same inspection, they encounter the same question and the same problem — but have no visibility into the fact that an action was already created yesterday for this exact item.

The improved platform solves this:

  • Existing action indicator — when conducting a recurring inspection, if a previous inspection (from the same template and site) has an open action on the same question, a visible badge is shown: "⚠ An open action already exists for this item." The user can tap to view the existing action's details, assignee, and status.
  • Link instead of duplicate — instead of creating a new action, the user can link the existing action to the current inspection, adding the new inspection as additional evidence. This prevents duplicate actions from piling up.
  • Create anyway — if the issue is genuinely different or the existing action doesn't cover it, the user can still create a new action.
  • Merge actions — if duplicate actions have already been created, users (with the right permissions) can merge two or more actions into one. Merging consolidates all linked inspections, comments, media, and activity into a single action. The merged (duplicate) actions are archived with a reference to the surviving action.
→ Connects to: Actions

Reports Output

Every completed inspection generates a report:

  • Export as PDF or Word — customisable report layouts control which info is shown/hidden.
  • Web report link — live, read-only URL accessible without an account.
  • AI summary — highlights critical findings, non-conformances, and required actions.
  • Multi-signature report section — all collected signatures are displayed with signer name, role, timestamp, and signature image.

Access Control Permissions

Template access levels: Conduct, Conduct & edit, Full access (archive/delete). Inspection access configurable per template: visible to inspector only, site members, specific groups, or the entire organisation. Access can be granted explicitly or dynamically based on group + site membership.

Issues

Module 02

The Issues module is a dedicated incident and observation reporting system. Any worker can capture observations, hazards, near-misses, and incidents the moment they occur, then route them to the right people for resolution.

Reporting an Issue — Step by Step Workflow

  • Step 1 — Choose the issue category. Select from pre-configured categories (e.g. "Near Miss", "Equipment Damage"). Each category has its own fields, questions, notification rules, and access settings.
  • Step 2 — Enter the issue title. Short descriptive name. Always required.
  • Step 3 — Complete the issue fields. Depending on the category: description, location (GPS), date occurred, site, photos/video/PDF, weather, custom questions. Fields are visible or hidden and required or optional per category.
  • Step 4 — Submit. Issue is logged with a timestamp. Automated notifications fire.

Issue Categories Configuration

Each category is independently configured: custom field visibility, custom questions (up to 10 per category, text or multiple-choice), category visibility controls, access rules ("Report only" / "Report + view" / full edit), notification rules (Private/Summary/Detailed), Critical Alerts (bypass DND), and linked templates (up to 25).

Issue QR Codes Contactless Reporting

Generate QR codes tied to a category and optionally pre-filled with a site. Anyone — even without an account — scans the code and fills out the issue form.

Issue Profile & Collaboration Management

Three tabs: Overview (summary, responses, location, linked items), Files (all attachments), Activity (real-time audit trail). Team members add comments, files, and update fields — all logged.

Starting an Inspection from an Issue Cross-module

From an issue, start a linked inspection (from pre-linked templates). Auto-linked back to the issue for full traceability.

→ Connects to: Templates & Inspections

Creating an Action from an Issue Cross-module

Create an action to assign resolution. Auto-linked to the issue; progress visible in the Overview tab.

→ Connects to: Actions

Investigations Root Cause

Create an investigation from an issue. Link multiple related issues, inspections, and media to uncover root causes. Own workspace, access controls, and audit trail.

Issue Reports, Views & Analytics Output

Export as public link or PDF. Custom views with filters (status, category, site, date, assignee). Analytics dashboards show trends by category, status, and resolution time.

Actions

Module 03

The task management engine. Actions ensure findings turn into follow-through — from quick fixes during inspections to formal corrective actions to recurring work orders.

Creating an Action — Where and How Workflow

Entry points: from an inspection question (notes/media carry over; logic can require it), from an issue (auto-linked), from an asset profile (asset/site pre-filled), from the maintenance table, from sensor alerts, or standalone (from the Actions list).

Fields: action type, title (255 chars), description (500 chars), assignee, priority, due date, site, labels, and custom fields.

Custom Action Types & Fields Configuration

Define types like "Corrective Action", "Work Order", "Preventive Maintenance", "General Task". Each type has its own required/custom fields (text, currency, date/time).

Priority & Auto-calculated Due Dates Configuration

Three priority levels. Configure auto-calculated due dates per priority (e.g. High = 24h, Medium = 7d, Low = 30d).

Recurring Actions Automation

Standalone actions can be set to recur on a schedule for routine tasks.

Custom Statuses & Transition Controls Workflow

Rename statuses (e.g. "Awaiting Review", "Approved", "Closed"). Configure transition controls — restrict which users can move actions to specific statuses, enforcing review gates.

Merge Actions Deduplication ✦ Improved

When duplicate actions exist (common with recurring inspections), users with the right permissions can merge two or more actions into one. Merging consolidates all linked inspections, comments, media, and activity into a single surviving action. Merged duplicates are archived with a reference link. This keeps the action list clean and avoids fragmented accountability.

Collaboration, Labels & Access Management

Built-in chat per action. Labels for grouping. Detail field editing controls. Actions from inspections inherit inspection access; standalone actions configurable (all users / site members / creator + assignee).

Linking Inspections to Actions Cross-module

Link templates to action types. Assignees can start an inspection from within an action. Up to 25 templates per action.

Heads Up

Module 04

One-to-many broadcast channel for reaching frontline workers. Consolidates workplace announcements into the same platform where teams do inspections and actions.

Creating a Heads Up Workflow

Fill in: Title (required), Description (required), Media (images, videos, PDFs), Assignees (individuals, groups, or sites). Then choose: Publish immediately, Save as draft, or Schedule for later.

Three Engagement Levels Accountability ✦ Improved

When creating a Heads Up, the admin configures which engagement levels they want from recipients. These are three independent options that can be combined in any way:

1. View Tracking (always on)

  • Every Heads Up automatically tracks who has viewed it and who hasn't.
  • The creator sees a real-time list of viewers with timestamps.
  • This is passive — recipients don't need to take any action; the system logs when they open the message.

2. Request Acknowledgement (optional toggle)

  • When enabled, recipients see an "Acknowledge" button after viewing the Heads Up.
  • Acknowledging confirms the recipient has read and understood the message.
  • The creator sees a real-time breakdown: Viewed vs. Acknowledged vs. Not yet viewed.
  • Recipients who have viewed but not acknowledged can receive targeted reminders.

3. Request Signature (optional toggle)

  • When enabled, recipients see a "Sign" prompt — they must provide a digital signature (name, signature image, timestamp).
  • Signing is a stronger commitment than acknowledging — it creates a compliance-grade record that the recipient has formally signed off on the content.
  • The creator sees a real-time breakdown: Viewed vs. Acknowledged vs. Signed vs. Not yet viewed.
  • Reminders can be sent specifically to recipients who haven't signed.
  • The Heads Up report includes a full signature log with each signer's name, timestamp, and signature image.
  • Signed Heads Ups are stored with signature data for auditing and regulatory purposes.

Combinations: An admin can enable just view tracking (informational broadcast), view + acknowledge (important update), view + acknowledge + sign (policy sign-off or safety briefing), or even view + sign without acknowledgement (formal sign-off only). The tracking dashboard adapts to show whichever levels are active.

Engagement Dashboard Monitoring

Every Heads Up has a real-time engagement dashboard showing:

  • Summary bar — at a glance: X viewed, Y acknowledged, Z signed, W not yet viewed.
  • Recipient list — filterable by status (viewed / acknowledged / signed / not viewed). Each row shows the recipient's name, group, site, and timestamps for each action they've taken.
  • Export — the full engagement data can be exported as CSV or PDF for compliance records.

Linking to Documents Cross-module

Create a Heads Up linked to a document from the Documents module. Users with file access are auto-added as assignees. Great for broadcasting new SOPs, policies, or safety bulletins alongside the document.

→ Connects to: Documents

Reminders, Comments & Reactions Engagement

Send reminders to non-responders (filterable: remind only those who haven't acknowledged, or only those who haven't signed). Recipients reply with comments and emoji reactions. Workers can reply/react but cannot create Heads Ups.

External Sharing, Editing & Analytics Management

Public links for external sharing. Edit or delete published Heads Ups. Export list to CSV. Engagement feeds into Analytics — charts show reach, acknowledgement rates, and signature rates independently.

Assets & Maintenance

Module 05

Digital register for every physical item — machinery, vehicles, safety gear, tools, facilities. Each asset has its own profile connecting inspection history, actions, maintenance, usage data, and location tracking.

Creating & Managing Assets Setup

Add individually or in bulk via CSV. Each record: name, type, site, photo, custom fields. Bulk CSV for onboarding or standardising data.

Custom Asset Types & Fields Configuration

Define types ("Generator", "Fire Extinguisher", "Delivery Truck") with custom fields (VIN, fuel type, warranty, serial number, etc.).

Nested Assets (Parent–Child Hierarchy) Structure ✦ Improved

Assets can be organised in a parent–child hierarchy. A main (parent) asset can have multiple sub-assets nested underneath it. For example:

  • A Building (parent) contains sub-assets: HVAC Unit, Fire Alarm Panel, Lift A, Lift B.
  • A Production Line (parent) contains sub-assets: Conveyor Motor, Packaging Machine, Labelling Station.
  • A Vehicle (parent) contains sub-assets: Engine, Gearbox, Hydraulic System.

When creating or editing an asset, the user can select a parent asset to nest it under. The parent asset's profile then displays all its sub-assets in a clear list. Sub-assets have their own full profiles (inspections, actions, readings, documents) just like any other asset.

Maintenance plans can be attached at either level:

  • Plan on the parent asset — e.g. a "Quarterly Building Safety Inspection" plan attached to the Building. Covers the parent as a whole.
  • Plan on a sub-asset — e.g. a "Monthly HVAC Filter Change" plan attached specifically to the HVAC Unit. Only applies to that sub-asset.
  • Both — a parent can have its own plans while each sub-asset also has independent plans. The maintenance table shows all plans in a flat view, with a column indicating whether the plan belongs to a parent or sub-asset, so nothing is hidden.

This hierarchy also applies to inspections and actions: when starting an inspection from a sub-asset, both the sub-asset and its parent are linked. Actions created on a sub-asset are visible from both the sub-asset's profile and the parent's profile (under a "Sub-asset actions" section), giving the parent-level manager full visibility across all components.

Nesting is one level deep (parent → sub-asset) to keep things simple. Sub-assets cannot have their own children.

Asset Profile Central Hub

Single source of truth: name/type/site/photo, linked inspections and actions, maintenance plans, readings over time, last known location, attached documents/images. For parent assets, the profile also displays a sub-assets list with quick status indicators (e.g. overdue maintenance, open actions). From the profile: start an inspection (pre-filled), create an action (pre-filled), view maintenance schedule.

QR Codes, Barcodes & RFID Field Access

Unique QR/barcode per asset. Scan with mobile app to instantly open the profile. RFID supported on Zebra devices.

Asset Readings & Utilisation Data

Track readings (odometer, runtime, temperature) manually via inspections or automatically via telematics (Samsara, Geotab, Caterpillar, AT&T Fleet). Utilisation charts show usage over time. Export to CSV.

Location Tracking Visibility

Inspection GPS updates last known location. Telematics provides real-time Live Map and Event History for past movements.

Preventative Maintenance Plans Automation ✦ Improved

The maintenance system has been simplified. Instead of the previous two-tier structure (programs containing plans), there is now a single concept: Maintenance Plans.

A maintenance plan defines when and how an asset should be serviced:

  • Plan name & description — e.g. "Oil Change", "Quarterly Safety Check", "Brake Pad Replacement".
  • Plan type — either usage-based (trigger at a set odometer/runtime reading) or time-based (trigger on a calendar interval, e.g. every 90 days).
  • Assigned assets — select which assets this plan applies to, filtered by type or site.
  • Last service details — record the most recent service reading or date to establish the baseline for calculating the next due date.
  • Notifications — up to 10 reminders per plan, alerting users/groups when maintenance approaches or is overdue.

The Maintenance Table gives a centralised view of every plan — on schedule, approaching, overdue, or missing data. From the table, create actions to assign servicing tasks. No need to navigate into a separate "program" layer first — everything is flat, visible, and actionable in one view.

Creating Actions & Inspections from Assets Cross-module

Create actions from any asset profile (pre-filled). Set recurring actions for ongoing tasks. Connect templates via the Asset question type for scheduled inspections.

Offline Access & Site Assignment Field Readiness

Works offline on mobile. Assets assigned to sites (individually or bulk CSV). Site membership controls visibility.

Documents

Module 06

A centralised, secure document management system that gives teams instant access to the files they need — SOPs, manuals, policies, safety bulletins, installation guides, and checklists — from any device, including offline.

Uploading & Organising Documents Setup

Upload files (PDF, Word, images, etc.) via the web app. Documents are organised into folders and subfolders with a main organisational folder structure. Labels can be applied for additional categorisation. Administrators with "Documents: Manage" permission can create and manage the folder structure.

Access Control Permissions

Document access is managed per file or folder. Control who can view, edit, and manage access to each document. Unlike sites (which are used for organisation/filtering in Documents, not access control), document access is granted explicitly to users or groups.

Site Assignment for Filtering Organisation

Documents can be assigned to sites for organisation and filtering purposes. This lets users filter the document library by location to quickly find files relevant to their site. Note: site assignment in Documents is for filtering only — it does not control view/edit access.

Offline Access & Sync Field Readiness

Documents are available offline on the mobile app. Workers in the field or in remote areas can access critical files (e.g. equipment manuals, safety procedures) without connectivity. Changes sync automatically when back online.

AI-Powered Answers Assistance

Workers can ask questions about documents and receive AI-generated answers drawn from the document content. This helps teams quickly find specific information within long documents without reading the entire file.

Linking Documents to Heads Ups Cross-module

From a document, create a Heads Up to broadcast it to the team. The document is attached, and users with file access are auto-added as assignees. This ensures that when a new SOP or policy is published, the right people are notified and can acknowledge or sign.

Linking Documents to Inspections & Assets Cross-module

Documents can be attached to asset profiles (manuals, warranties, service records) and referenced within inspection templates (as instruction fields or reference material). This keeps critical documentation accessible at the point of use.

Analytics

Module 07

Analytics sits across the entire platform, pulling data from every module into customisable dashboards and charts. It transforms operational data into actionable insights — helping teams spot trends, track KPIs, and make data-driven decisions.

Dashboards Core Concept

A dashboard is a collection of charts on a single screen, focused on a specific theme (e.g. "Safety Performance", "Maintenance Compliance", "Issue Trends"). Users can create multiple dashboards. Each dashboard can be shared with specific users, groups, or the entire organisation.

Dashboard Templates Getting Started ✦ Improved

To make it easier to get started, the platform provides pre-built dashboard templates:

  • Inspection Performance — completion rates, overdue inspections, flagged response trends, completion by site/user.
  • Issue Management — issue volume by category, resolution times, open vs. closed, issues by site.
  • Action Tracking — actions created vs. completed, overdue actions, average close time, breakdown by type/priority.
  • Heads Up Engagement — reach, acknowledgement rates, signature rates, engagement by site/group.
  • Asset & Maintenance — maintenance compliance, overdue plans, asset utilisation, readings over time.
  • Compliance Overview — overall compliance score, non-compliant rules, due-soon rules, score by category and site, trend over time, goal progress.
  • Team Compliance — cross-module view combining inspections, training, and issue response metrics per site.

Each template comes pre-configured with relevant charts and filters. Users can use them as-is or customise them (add/remove/edit charts, change filters, rename).

Charts Visualisation

Charts are the building blocks of dashboards. Each chart has:

  • Data source — select from inspections, issues, actions, Heads Up, assets, maintenance, or training.
  • Metric — what to measure (e.g. count, percentage, average time, score).
  • Grouping — how to break down the data (by site, user, group, template, category, type, priority, time period, asset, asset type).
  • Chart type — bar, line, pie, donut, table, or number card.
  • Filters — narrow data by date range, site, group, user, template, category, status, priority, asset type, or any combination.

Charts update in real-time as new data is captured across the platform.

Filters Precision

Filters can be applied at the dashboard level (affecting all charts) or at the individual chart level. Key filter dimensions include: date range, site (with hierarchy drill-down — e.g. Country → Region → Site), group, user, template, issue category, action type, action priority, asset, and asset type. Filters can be saved as part of the dashboard configuration.

Data Export Output ✦ Improved

All analytics data can be exported:

  • CSV export — export the underlying data of any chart or entire dashboard as a CSV file for further analysis in Excel, Google Sheets, or BI tools.
  • PDF export — export a dashboard as a formatted PDF report for sharing with stakeholders.
  • Scheduled exports — configure automatic recurring exports (e.g. weekly CSV emailed to a distribution list).

Cross-Module Analytics Coverage

Analytics can pull data from every module:

  • Inspections — completion rates, flagged responses, overdue counts, response distributions.
  • Issues — volume by category/site, resolution times, open/closed ratios, trend lines.
  • Actions — created vs. completed, on-time percentage, average close time, breakdown by type/priority.
  • Heads Up — reach, acknowledgement rates, signature rates, engagement by audience.
  • Assets — utilisation, reading trends, maintenance compliance, overdue plans.
  • Compliance — compliance scores by category/site, non-compliant rule counts, trend lines, goal progress.
  • Training — course completion, quiz scores, certification status.

Drill-Down & Action from Analytics Workflow

Charts are interactive. Users can tap or click to drill down into the underlying data — e.g. click a bar showing "12 overdue actions at Manchester" to see the specific actions. From the drill-down, users can take action directly: update an action, start an inspection, or create a Heads Up. This closes the loop between insight and action.

Compliance

Module 08

A simple, evidence-based compliance engine. Set up the frameworks your organisation must adhere to — ISO 45001, ISO 9001, internal quality standards, or anything custom — define the rules within each, and the platform automatically collects proof from inspections, actions, training, documents, and more. At a glance, see whether you're compliant, where the gaps are, and what needs to happen next. Inspired by platforms like Vanta, but radically simplified for operational teams.

Core Concept: Frameworks → Rules → Evidence → Status How It Works ✦ New Module

The Compliance module is built around one simple hierarchy:

  • Frameworks — you define which standards, certifications, or regulatory requirements your organisation needs to comply with (e.g. "ISO 45001", "ISO 9001", "UK HSE Regulations", "Internal Quality Manual"). A framework is a named container that holds a set of related rules.
  • Rules — within each framework, you define the specific things your organisation must do (e.g. "Monthly fire safety audit at every site", "Risk assessments reviewed quarterly"). Each rule belongs to exactly one framework.
  • Evidence — for each rule, you specify what counts as proof. The platform auto-collects evidence from activities already happening in other modules (inspections, actions, training, documents, Heads Ups). No extra work for the frontline.
  • Status — the platform continuously evaluates whether evidence exists and is current. Each rule shows a live status: Compliant, Due Soon, or Non-Compliant. The framework inherits an overall score from the status of all its rules.

That's it. Frameworks group rules. Rules define requirements. Evidence proves compliance. Status is calculated automatically.

Creating a Framework Setup

A framework represents a standard, certification, or set of requirements the organisation wants to track compliance against. To create one:

  • Framework name — e.g. "ISO 45001:2018", "ISO 9001:2015", "UK HSE Fire Safety Regulations", "Company Quality Manual v3".
  • Description — a summary of what this framework covers and why the organisation follows it.
  • Type — categorise the framework: Health & Safety, Quality, Environmental, Regulatory, or Custom.
  • Owner — the user or group responsible for maintaining this framework (e.g. "QHSE Manager", "Quality Team").
  • Applicable sites — which sites this framework applies to. Can be all sites, a level in the hierarchy, or individually selected. Rules within the framework inherit this by default (but can be overridden per rule).

Once created, the framework appears on the Compliance dashboard with its own compliance score. The user then populates it with rules — either manually or by importing a pre-built rule template pack.

An organisation can have multiple frameworks active simultaneously. For example, a manufacturing company might have ISO 45001 (H&S), ISO 9001 (Quality), and ISO 14001 (Environmental) all running in parallel, each with their own set of rules and independent compliance scores.

Creating a Rule (within a Framework) Workflow

Rules live inside a framework. To create a compliance rule, the user selects the framework and fills in:

  • Rule name — a clear, human-readable name (e.g. "Monthly Fire Safety Audit").
  • Clause / Reference — optional. The specific clause or section of the standard this rule maps to (e.g. "Clause 9.1.2", "Regulation 3(1)(a)"). This keeps a clear link between the rule and the source requirement.
  • Description — what this rule requires and why it exists.
  • Frequency — how often the rule must be satisfied: one-time, weekly, monthly, quarterly, bi-annually, annually, or a custom interval.
  • Applicable sites — inherits from the framework by default. Can be narrowed to specific sites if the rule only applies to certain locations.
  • Responsible party — the user, group, or role accountable for ensuring this rule is met.
  • Evidence requirements — what constitutes proof (see below).
  • Grace period — optional. Days after the due date before the rule is marked Non-Compliant.

Evidence Types — Auto-Collected from the Platform Cross-module

This is where the Compliance module connects to the entire ecosystem. For each rule, the user defines one or more evidence requirements. Evidence is auto-collected — the platform watches for matching activities and links them to the rule automatically. Evidence types include:

  • Inspection completed — "A completed inspection from template [X] at the applicable site within the period." The platform auto-detects when a matching inspection is completed and marks the evidence as collected.
    → From: Templates & Inspections
  • Action completed — "A completed action of type [Y] at the applicable site within the period." Auto-detects when a matching action is marked complete.
    → From: Actions
  • Training completed — "All users in group [G] have completed course [Z] within the period." Auto-detects training completion.
  • Document current — "Document [D] has been reviewed/updated within the last [N] days." Auto-checks the document's last-modified date.
    → From: Documents
  • Heads Up signed — "Heads Up [H] has been signed by all assignees." Auto-checks the signature status.
    → From: Heads Up
  • Asset maintenance current — "All assets of type [T] at the applicable site have no overdue maintenance plans." Auto-checks the maintenance table.
    → From: Assets & Maintenance
  • Issue resolution SLA met — "All issues in category [C] at the applicable site were resolved within [N] days." Auto-checks issue resolution times.
    → From: Issues
  • Manual evidence — for things that happen outside the platform (e.g. third-party audit certificates, government approvals, insurance documents). The responsible user uploads a file (PDF, image) and enters a validity date. The platform tracks the expiry and alerts when renewal is due.

A single rule can require multiple evidence types. For example, "ISO 45001 — Emergency Preparedness" might require: (1) a completed Emergency Drill inspection, (2) all staff completed Fire Safety training, and (3) the Emergency Response Plan document reviewed within the last 12 months. All three must be satisfied for the rule to be Compliant.

Compliance Status — Per Rule, Per Site, Per Period Tracking

The platform continuously evaluates every rule against its evidence requirements. Status is calculated per rule, per applicable site, per time period:

  • ● Compliant — all required evidence is present and current for this period.
  • ● Due Soon — the current period's evidence deadline is approaching (configurable threshold, e.g. 7 days before due). Alerts are sent to the responsible party.
  • ● Non-Compliant — evidence is missing, expired, or the grace period has passed. Escalation notifications fire.

Status updates are automatic and real-time. When an inspector completes an inspection, the platform immediately checks whether it satisfies any pending compliance rule — and if it does, the rule's status flips to Compliant without anyone needing to manually log or link it.

Compliance Dashboard Visibility

A single-screen view showing the organisation's compliance health:

  • Framework cards — each active framework is displayed as a card showing its name, type, overall compliance score, and a mini progress bar. Click to drill into the framework's rules. This is the primary entry point — the user sees at a glance which frameworks are healthy and which need attention.
  • Overall compliance score — aggregated across all frameworks (e.g. "87% compliant across 3 frameworks, 42 rules").
  • Breakdown by framework — each framework with its own score, number of compliant/due/non-compliant rules.
  • Breakdown by site — compliance score per site, with the site hierarchy allowing roll-up views (e.g. all of "UK Region" at 91%, with individual sites underneath).
  • Non-compliant rules list — prioritised across all frameworks, sorted by how long overdue. Each row shows the framework, rule name, clause reference, affected site(s), missing evidence, and responsible party.
  • Due soon list — rules approaching their deadline, so teams can act proactively.
  • Trend line — compliance score over time (per framework or aggregated), showing improvement or decline.

Notifications & Escalations Automation

The Compliance module sends automated notifications at key moments:

  • Due soon reminder — sent to the responsible party when a rule's evidence deadline is approaching (configurable: 14 days, 7 days, 3 days, 1 day before).
  • Overdue alert — sent when a rule becomes Non-Compliant.
  • Escalation — if a rule remains Non-Compliant beyond a configurable escalation threshold (e.g. 7 days overdue), a notification is sent to the responsible party's manager or a designated escalation group.
  • Compliance restored — sent when evidence is collected and the rule returns to Compliant status.

Compliance Goals Targets

Optionally, administrators can set compliance goals — per framework, per site, or organisation-wide. For example: "Achieve 95% compliance on ISO 45001 across all UK sites by Q3." Goals are tracked on the dashboard with a progress indicator showing current score vs. target and a projected trajectory based on recent trends.

Compliance Report — Auditor-Ready Output

Generate a Compliance Report for external auditors, regulators, or internal review:

  • Select a specific framework (e.g. "ISO 45001:2018") or generate a cross-framework report.
  • Lists all rules within the selected framework, their status, clause references, and the evidence linked to each.
  • Filterable by site, time period, and status.
  • For each rule, the report shows: the clause/reference, requirement description, frequency, applicable sites, responsible party, all linked evidence (with links to the actual inspections, training records, documents, etc.), and a compliance history timeline.
  • Exportable as PDF (formatted for auditors) or CSV (for data analysis).

During an ISO audit, for example, the user selects the "ISO 45001:2018" framework, exports the report, and hands it to the auditor — every clause mapped to a rule, every rule mapped to evidence, with full traceability.

Framework Templates Getting Started

To make setup fast, the platform provides pre-built framework templates — complete frameworks with rules already mapped to the relevant clauses:

  • ISO 45001:2018 (Occupational Health & Safety) — key clauses mapped to rules with suggested evidence types.
  • ISO 9001:2015 (Quality Management) — key clauses mapped to rules.
  • ISO 14001:2015 (Environmental Management) — key clauses mapped to rules.
  • Local H&S regulations — country-specific framework packs (e.g. UK HSE, OSHA).
  • Blank framework — an empty framework for custom company policies or internal standards.

A framework template is imported with one click, creating the framework and all its rules instantly. The user then customises: adjust frequencies, assign sites, link evidence types to the organisation's actual templates, courses, and documents. This gets teams from zero to a working compliance system in minutes, not weeks.

How Compliance Connects to Every Module Ecosystem

The Compliance module is not a standalone silo — it sits on top of the entire platform, organised by framework, and draws evidence from every module:

  • Templates & Inspections — completed inspections auto-satisfy inspection-based rules.
  • Issues — issue resolution times are checked against SLA-based rules.
  • Actions — completed actions of specific types satisfy action-based rules.
  • Heads Up — signed Heads Ups satisfy briefing/sign-off rules.
  • Assets & Maintenance — maintenance plan compliance is checked for asset-based rules.
  • Documents — document freshness is checked for document-based rules.
  • Training — course completion across groups satisfies training-based rules.
  • Analytics — compliance scores and trends feed into Analytics dashboards alongside operational data. A "Compliance" data source is available in the chart builder, filterable by framework.
  • Groups & Sites — frameworks are scoped to sites; responsibilities assigned to groups. Site hierarchy enables roll-up compliance views across regions.

The result: teams don't need to do extra work for compliance. They keep doing their inspections, actions, training, and maintenance — and the Compliance module watches, collects evidence, and keeps score automatically across every active framework.

Groups & Sites

Module 09

The organisational backbone. Groups control people and access. Sites control data and location. Together, they form the foundation every other module builds on.

Groups vs. Sites Concept

Groups organise users by team, role, or department. Primary purpose: manage access (templates, inspections, training, notifications). Sites organise data by location, project, or branch. Primary purpose: tag and filter operational data (inspections, actions, issues, assets). Users can belong to multiple groups and sites. Combined via advanced access rules for precise control.

Creating & Managing Groups Setup

Create groups via web app. Add users manually, via CSV/Excel, or via rule-based membership (auto-add/remove based on custom user fields like role, department, shift). Up to 5 rules per group, 15,000 users per rule-based group, 100 groups per user.

How Groups Control Access Cross-module

Groups are a target for access and assignment across every module: template access, training assignment, issue notifications, Heads Up targeting, action notifications, and lone worker escalations.

Creating Sites & Site Hierarchy Setup

Add individually or bulk CSV. Up to 50,000 sites. Organise into a hierarchy (Country → Region → Area → Site). Higher levels aggregate data and membership. Manage members manually, via Site Matrix, CSV, or membership rules. Custom site labels to match organisational terminology.

Site Profiles Central View

Each site has a profile aggregating: members, inspections, actions, issues, and assets linked to that site.

How Sites Control Data Cross-module

Inspections tagged to sites for filtering and access. Actions linkable to sites. Issues reportable per site. Assets assigned to sites for visibility. Heads Up targetable by site. Schedules assignable per site. Analytics filterable by site/hierarchy. Documents filterable by site (not access-controlled).

Advanced Access Rules (Groups + Sites) Permissions

Combine group + site membership: e.g. "Grant access to users in 'Auditors' group AND members of 'Manchester' site." Available for templates, inspections, issues, actions, and training.

Site Selection Settings & Matrix Admin

Configure whether users see all sites or only their own. The Group & Site Matrix is a grid for managing memberships at a glance, exportable as CSV.

Settings

Module 10

The Settings page is the central administration hub. What a user sees here depends on their permission set — administrators see everything; standard users see only their personal settings.

Pricing Model Concept ✦ Improved

The platform uses a per-company pricing model. There are no individual seat types or per-user charges. A company pays a single price for access to the platform. This means:

  • No concept of "full seat" vs. "lite seat" vs. "guest seat". All users within a company have access to the platform.
  • What a user can do is controlled entirely by their permission set and group membership — not by how much the company pays per head.
  • Companies can add as many users as they need without worrying about per-seat cost implications.
  • This simplifies onboarding: add a user, assign them to the right group(s) and site(s), give them the right permission set, and they're ready to go.

Permission Sets What Users Can Do

Permission sets are bundles of granular permissions controlling which features a user can access. Default sets include:

  • Administrator — full control: users, groups, sites, billing, all features.
  • Manager — create templates, manage training, manage feature settings.
  • Standard — conduct inspections, report issues, complete actions and training.

Custom permission sets can be created with any combination of granular permissions (e.g. "Actions: Settings", "Issues: Settings", "Assets: Manage", "Heads Up: Manage", "Analytics: Create dashboards", etc.). Every user must have exactly one permission set.

Key distinction: Permission sets control what features a user can use. Groups and sites control what data they can see. Both work together.

User Management Admin

From Settings, administrators can:

  • Add, invite, deactivate, or reactivate users.
  • Assign or change permission sets (individually or in bulk via CSV).
  • Add users to groups and sites (individually, bulk CSV, or rule-based).
  • Manage custom user fields (role, department, location, etc.) used for rule-based membership.
  • Reset passwords and manage account credentials.
  • View and download the full user list as CSV.

Organisation Settings Admin

Organisation-level settings include:

  • Groups — create, edit, manage memberships and rules.
  • Sites — create, edit, manage hierarchy and membership.
  • Permission sets — create and edit permission bundles.
  • Templates — manage Global Response Sets and default template settings.
  • Actions — manage action types, custom fields, statuses, transition controls, labels, priority-based due dates, and access rules.
  • Issues — manage categories, custom questions, notifications, Critical Alerts, QR codes, and access rules.
  • Assets — manage asset types, fields, readings, maintenance plans, and settings.
  • Heads Up — manage default settings.
  • Compliance — manage frameworks, rule templates, notification thresholds, escalation policies, and goal targets.
  • Training — manage courses, quizzes, and assignments.
  • Integrations — connect to third-party tools (telematics, ERP, BI, etc.).
  • Billing — manage the company subscription.

Personal Settings (All Users) Self-service

Every user can access their own personal settings:

  • Edit name, email, and password.
  • Set timezone and language preferences.
  • Manage notification preferences (email, push, in-app).
  • View their group and site memberships.
  • View their assigned permission set (read-only).

How the Modules Interact

Inspection → Action (with deduplication)

A user conducts a daily inspection and flags a broken guardrail. They create an action from the question. The next day, during the same recurring inspection, the platform shows a badge: "An open action already exists for this item." The user links the existing action instead of creating a duplicate. If duplicates were already created, a manager merges them into one.

Templates & Inspection → Actions (dedup + merge)

Issue → Inspection → Action

A worker reports a hazard via Issues. A manager opens the issue, starts a linked inspection for detailed assessment. During the inspection, they create an action for the fix. The issue profile shows both linked items — full traceability.

Issues → Templates & Inspection → Actions

Asset → Inspection → Maintenance Plan

A technician scans an asset's QR code, opens the profile, starts a scheduled inspection (pre-filled). They log an odometer reading that triggers a maintenance plan threshold. They create an action (Work Order) from the maintenance table.

Assets → Templates & Inspection → Maintenance → Actions

Parent Asset → Sub-Asset Maintenance

A production line (parent asset) has three sub-assets: Conveyor Motor, Packaging Machine, and Labelling Station. Each sub-asset has its own maintenance plan (e.g. monthly lubrication for the motor, quarterly calibration for the labelling station). The parent also has a plan (annual safety audit). A manager views the parent asset's profile and sees all sub-assets with status indicators — the motor is overdue. They drill into the sub-asset, create an action (Work Order), and assign it. The parent profile shows the open action under "Sub-asset actions".

Parent Asset → Sub-Assets → Maintenance Plans → Actions

Multi-Signature Inspection → Approval

A work permit template requires 3 signatures in sequence: the inspector, the site supervisor, and the safety manager. The inspector completes the form and signs. The supervisor is notified, reviews, and signs. The safety manager signs last. Only then can the inspection be completed and the work permit issued as a report.

Templates & Inspection (multi-signature + approval)

Issue → Action → Heads Up (with signature)

An incident is reported via Issues. An action resolves it. A manager creates a Heads Up with updated safety procedures, attached as a Document, and requests signatures. All workers sign the Heads Up, and the signature log is stored for compliance.

Issues → Actions → Documents → Heads Up (signature)

Documents → Heads Up → Analytics

A new SOP is uploaded to Documents. A Heads Up is created linking the document and requesting acknowledgement. Analytics tracks which sites and groups have acknowledged and which haven't, showing a completion heat map by region.

Documents → Heads Up → Analytics

Analytics → Drill-Down → Action

A manager views the Analytics dashboard and sees 15 overdue actions at a specific site. They drill down to see the individual actions, update priorities, and send a Heads Up to the site team about the backlog.

Analytics → Actions → Heads Up

Compliance — Automatic Evidence Collection

An organisation has an ISO 45001 framework with a rule: "Monthly Fire Safety Audit at every site." The rule is linked to a specific inspection template as evidence. When a site manager completes the fire safety inspection in January, the Compliance module auto-detects the completed inspection, matches it to the rule and site, and flips the status to Compliant for January. The ISO 45001 framework score ticks up. In February, the rule shows Due Soon — the site manager receives an automated reminder. If March arrives without an inspection, the status flips to Non-Compliant and an escalation fires to the regional safety lead.

Templates & Inspection → Compliance Framework (auto-evidence) → Notifications

Compliance — Multi-Evidence Rule for ISO Audit

Inspections + Training + Documents → Compliance Framework → Report

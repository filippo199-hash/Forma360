01 — Templates & Inspections

Edge Cases & Resolutions
IDEdge CaseResolution
T-E01A signer in a sequential multi-signature chain is deactivated or removed from the org while the inspection is waiting for their signature.The inspection flags the blocked signature slot. The inspection creator (or an admin) is notified and can reassign the slot to a different user. The inspection remains in "Awaiting Signatures" until resolved.
T-E02The same user is assigned to multiple signer slots in a multi-signature flow.Prevent at template design time — validate that no duplicate users appear across signer slots. If slots are "select at inspection time", validate at signing time that the same user cannot be selected twice.
T-E03Two users start the same scheduled inspection offline and both sync when reconnected.Both inspections are saved as separate records. The system flags the duplicate on sync and notifies the schedule owner, who can archive one or keep both. The schedule is marked as fulfilled by the first synced inspection.
T-E04A template is edited (questions added/removed) while inspections are in progress.Template changes only apply to new inspections. In-progress inspections continue with the version of the template they were started with. A version number is stored per inspection.
T-E05A template is archived or deleted while inspections are in progress.In-progress inspections can still be completed. No new inspections can be started from the archived template. Completed inspections and reports remain accessible. Schedules linked to the template are paused with a notification to the admin.
T-E06An approval page approver is deactivated while an inspection is pending their approval.The inspection is flagged as "Approval blocked". An admin is notified and can reassign the approval to another user. The inspection stays in "Pending Approval" state.
T-E07Logic nesting exceeds 40 levels during template design.The template editor blocks further nesting at 40 levels with a clear error message. Existing logic is preserved.
T-E08A required question is inside a logic branch that is never triggered during an inspection.Required questions inside untriggered logic branches are not enforced — the inspection can be completed without answering them. The system only enforces required status on visible questions.
T-E09The inspection title format produces a string exceeding 250 characters.Truncate the title at 250 characters. When exporting reports, file names are also trimmed. Warn the template creator if the format is likely to produce long titles.
T-E10A user tries to add a 101st row to a table during an inspection.Display a clear message: "Maximum 100 rows reached." The "Add row" button is disabled. The user can delete a row to add a new one.
T-E11Multi-signature inspection conducted offline — other signers aren't available offline.The inspector completes and signs their part offline. The inspection syncs when online. The remaining signers are then notified. The inspection stays in "Awaiting Signatures" until all sign online.
T-E12Action deduplication: the "existing action" indicator matches an action from a different site using the same template.Deduplication matching is scoped to the same template + same site + same question. Actions at different sites are not surfaced as duplicates.
T-E13A recurring inspection's template is updated between occurrences — old flagged questions no longer exist.Each inspection records the template version. Deduplication compares actions by question ID. If the question ID no longer exists in the new template version, no dedup indicator is shown.
T-E14The user converts a malformed PDF/Excel into a template — structure is broken.The conversion is a best-effort process. Show a preview for the user to review before saving. If parsing fails, display an error and suggest manual template creation. Never auto-publish a converted template.
T-E15A scheduled inspection is assigned to a group, but a user is removed from the group after the schedule is published.The removed user no longer receives schedule reminders or overdue notifications. Their in-progress inspections (if any) remain accessible for completion but the next occurrence won't be assigned to them.
T-E16Calendar drag-and-drop rescheduling moves a recurring inspection — does it affect all future occurrences or just one?Prompt the user: "Reschedule this occurrence only" or "This and all future occurrences." Default to single occurrence.
T-E17A Global Response Set is edited — what happens to all templates using it and in-progress inspections?Templates using the Global Response Set are updated to the new version on next publish. In-progress inspections retain the version they were started with. Completed inspections are unaffected — historical data is preserved.
T-E18Two users with edit access open the same template simultaneously and both make changes.Last-save-wins with conflict detection. When the second user saves, they are warned: "This template was modified by [User] at [time]. Review their changes before saving." Show a diff of changes. The user can merge or overwrite.
T-E19An inspection contains 50+ high-resolution photos — PDF export is very large.Photos are compressed for PDF export (configurable quality setting). If the export exceeds a size threshold (e.g. 100 MB), it runs as a background job with a download link. A warning is shown during export.
T-E20Parallel signers attempt to sign at the exact same moment — race condition.Each signature is an atomic write operation. Both signatures are accepted independently. No conflict — parallel signing is designed for concurrent access. Timestamps reflect the actual moment each signed.

02 — Issues

Edge Cases & Resolutions
IDEdge CaseResolution
I-E01A user scans an Issue QR code without an account. What identity data is captured?Anonymous submissions capture the data entered in the form fields only (no user identity). The issue is created with "Reported by: Anonymous (QR)". Optionally, the form can include a "Your name" and "Your email" field.
I-E02An issue category is deleted while issues exist in it.Prevent deletion if open issues exist. Allow archiving instead — archived categories can't receive new issues, but existing issues remain accessible and filterable under the archived category name.
I-E03A custom question's response options are changed after issues have been reported with the old options.Existing issues retain their original responses. The change only applies to new issues. Historical data is preserved as-is.
I-E04A Critical Alert fires but the recipient has no mobile device registered.Fall back to email notification with "[CRITICAL]" prefix in the subject line. Log the failed mobile delivery in the notification audit trail.
I-E05An issue is reported offline — location shows coordinates instead of a full address.When the device reconnects, the platform reverse-geocodes the coordinates to a full address. Until then, coordinates are displayed.
I-E06A user with "Report only" access tries to view or edit the issue they reported.The user cannot view the issue after submission. They see a confirmation message with a reference number. To view, they'd need elevated access.
I-E07An investigation links issues from different sites — access rules conflict.Investigation access is independent of issue access. The investigation creator controls who can see the investigation. Linked issues are shown in summary form; full issue access still follows the issue's own rules.
I-E08Category visibility is changed to exclude a user after they've already reported issues in that category.The user's existing issues remain. They can no longer report new issues in that category. Their access to view existing issues depends on the issue access setting, not category visibility.
I-E09A user tries to upload a file exceeding the size limit when reporting an issue.Display a clear error: "File exceeds the maximum size of X MB. Please reduce the file size." The issue form retains all other entered data.
I-E10The user starts an inspection from an issue but the linked template has been archived since the category was configured.The archived template is greyed out with a label "Template archived". The user can select another available template. Admin is notified to update the category's linked templates.
I-E11An Issue QR code is scanned for an archived category.Display: "This reporting category is no longer active. Contact your administrator." The QR code does not open a form.
I-E12Multiple users report the same incident independently (near-duplicate issues).No automatic dedup for issues (unlike actions) — each report is valuable as independent evidence. However, the issue list shows a proximity indicator: "X other issues reported at this location in the last 24 hours." Managers can link related issues to the same investigation.

03 — Actions

Edge Cases & Resolutions
IDEdge CaseResolution
A-E01Merging actions with different assignees — who becomes the assignee of the merged action?The user performing the merge selects which assignee to keep. The other assignees are added as collaborators. All parties are notified.
A-E02Merging actions of different types (e.g. "Corrective Action" and "Work Order").The user performing the merge selects which type the surviving action should have. A warning is shown that the type will change for merged items.
A-E03Merging actions linked to different sites.Allow the merge but require the user to confirm the surviving action's site. Warn that the action will only be associated with one site post-merge.
A-E04A user creates a recurring action from an inspection (not a standalone action).Prevent this — the "Recur" option is only available on standalone actions. For inspection-linked actions, the recurrence is handled by the inspection schedule.
A-E05Priority is changed after creation when priority-based due dates are configured — does the due date recalculate?Yes, changing priority recalculates the due date from the creation date using the new priority's rule. The user is shown a confirmation: "Due date will change to [X]. Proceed?"
A-E06All status transitions are configured to block the current user — the action is stuck.An admin override exists: users with "Actions: Settings" permission can always transition any action regardless of transition controls. Log the override in the audit trail.
A-E07The action assignee is deactivated.The action's status doesn't change, but a "No active assignee" warning is shown. The action creator and admins are notified. The action can be reassigned.
A-E08An action is linked to an inspection that is subsequently deleted.The action remains but the inspection link shows "Inspection deleted". The action retains any data (notes, photos) that was copied from the inspection at creation time.
A-E09A custom field is marked as required on an action type, but existing actions of that type don't have it filled.Required field enforcement only applies to new actions. Existing actions are not retroactively blocked. A report can identify actions with missing required fields.
A-E10A recurring action generates a new instance while the previous one is still open.The new instance is created regardless. The previous open instance remains. A warning indicator shows "Previous occurrence still open" on both actions.
A-E11An action is linked to both an inspection AND an issue — which access rules apply?The action inherits the most permissive access. Users with access to either the inspection or the issue can see the action. The action's activity log shows both source links.
A-E12A label is deleted while actions have it applied.The label is removed from all actions silently. Actions are not affected otherwise. A confirmation prompt warns: "This label is applied to X actions. Removing it will untag them."

04 — Heads Up

Edge Cases & Resolutions
IDEdge CaseResolution
H-E01A user is added to a group after a Heads Up targeting that group was published.The new user does NOT see the Heads Up. Assignees are locked at the time of publishing. If the creator wants to include new members, they can edit the Heads Up and re-add assignees.
H-E02A user is removed from a group after a Heads Up was published but before they acknowledged/signed.The user retains access to the Heads Up (they were an assignee at publish time). They can still acknowledge/sign. Their status remains in the tracking dashboard.
H-E03A Heads Up is edited after some users have already signed. Are existing signatures invalidated?Warn the editor: "X users have already signed. Editing content will invalidate their signatures and they'll need to re-sign." The editor can proceed or cancel. If they proceed, all signatures are cleared and re-requested.
H-E04A signature is requested from a user on a device without touch capability (desktop browser).Provide a mouse-based signature pad (draw with cursor) and a "Type signature" fallback where the user types their name and it renders in a signature font. Both capture timestamp and user identity.
H-E05An external viewer (via public link) is asked to sign.External links provide view-only access. Signatures require a platform account. The external viewer sees a message: "Sign in to sign this Heads Up."
H-E06A Heads Up is scheduled for a future time, but the creator's account is deactivated before it publishes.The scheduled Heads Up is paused and flagged. An admin is notified. The admin can either publish it, reschedule, or delete it.
H-E07A Heads Up is sent to 10,000+ recipients. Engagement tracking at scale.Tracking data is aggregated and paginated. The dashboard shows summary counts instantly; the full recipient list loads progressively. Exports handle large datasets in background with a download link sent via notification.
H-E08A user views a Heads Up offline — can their view/acknowledgement/signature be captured?If the Heads Up was cached before going offline, the user can view it. Acknowledgement and signature are queued locally and synced when back online. View timestamp reflects the offline view time.
H-E09Admin enables all three engagement levels: does the user see View + Acknowledge + Sign as three separate steps, or is it combined?View is passive (auto-tracked). Acknowledge and Sign are shown as two sequential buttons: "Acknowledge" appears first. After acknowledging, "Sign" appears. The user can skip straight to Sign if both are enabled (signing implies acknowledgement).
H-E10A linked document is updated after the Heads Up was published — do recipients see the old or new version?The Heads Up links to the live document, not a snapshot. Recipients always see the current version. If the content has materially changed, the admin should re-publish or edit the Heads Up and consider re-requesting signatures (which triggers the invalidation warning per H-E03).
H-E11An admin deletes a Heads Up that has collected signatures — compliance implications.Warn: "This Heads Up has X signatures. Deleting it will remove these compliance records." Offer "Archive" as an alternative (hides from feed but preserves all data). If deletion is confirmed, signature data is retained in an audit log for 90 days before permanent deletion.

05 — Assets & Maintenance

Edge Cases & Resolutions
IDEdge CaseResolution
AS-E01A parent asset is deleted — what happens to its sub-assets?Prevent deletion if sub-assets exist. Offer two options: (1) "Delete parent and promote sub-assets to standalone" — sub-assets become top-level assets. (2) "Delete parent and all sub-assets" — requires explicit confirmation and admin permission.
AS-E02A sub-asset is moved to a different parent. Its maintenance plans reference the old parent's site.Moving a sub-asset prompts: "Update site to match new parent?" If yes, site updates and maintenance plans recalculate. If no, site stays — the sub-asset can be at a different site than its parent.
AS-E03A maintenance plan is attached to a parent — does it automatically apply to sub-assets?No. Parent plans and sub-asset plans are independent. The parent's plan covers the parent as a whole. Sub-assets need their own plans. The parent profile shows a consolidated view of all plans (own + sub-assets').
AS-E04A QR code is scanned for a deleted/archived asset.Display a message: "This asset has been [deleted/archived] and is no longer active. Contact your administrator." For archived assets, optionally show read-only historical data.
AS-E05Bulk CSV upload contains duplicate asset names.Allow duplicate names (different assets can share a name). Uniqueness is enforced by internal asset ID, not name. Flag duplicates in the upload review screen with a warning: "X assets share the same name — verify these are separate assets."
AS-E06Telematics integration disconnects — location data becomes stale.Show a "Last updated: [timestamp]" indicator on the Live Map. After a configurable threshold (e.g. 24 hours of no data), show a "Stale data" warning on the asset. An admin notification fires.
AS-E07A usage-based maintenance plan exists but no readings have been submitted.The plan shows status "Awaiting first reading." No due date is calculated until a baseline reading exists. A reminder is sent to the responsible party to submit the first reading.

X-E01A user is deactivated who is simultaneously: an action assignee, a multi-signature signer on a pending inspection, a Heads Up recipient who hasn't signed, the responsible party on 3 compliance rules, and an approver on an inspection awaiting approval.On deactivation, a cascade impact summary is shown listing every affected item across all modules. X-E02A site is archived — cascading impact across inspections (tagged to the site), actions (linked to site), issues (reported at site), assets (assigned to site), compliance rules (scoped to site), scheduled inspections (assigned to site), and Heads Ups (targeted to site members).Show a full cascade preview before archiving: "This site has X inspections, Y actions, Z issues, W assets, V compliance rules." All data retains the archived site reference for historical reporting. No new inspections/issues/actions can be tagged to the archived site. Schedules are paused. Compliance rules scoped only to this site become "Not Evaluable" (multi-site rules simply exclude this site from future periods). Assets remain in the register but lose their site filter.
X-E03A notification storm — a user receives 50+ notifications in rapid succession from different modules (overdue inspections, new issues, action assignments, Heads Ups, compliance alerts) all triggering at the same time.Implement notification batching. If more than N notifications fire for the same user within a 5-minute window, they are batched into a single digest notification: "You have X new items requiring attention" with a grouped summary. The user's notification preference settings control thresholds per channel (push vs. email vs. in-app).
X-E04An inspection is completed that simultaneously satisfies: an action dedup check (same template + site + question has an open action), a compliance rule (evidence auto-collected), an asset reading threshold (triggers maintenance plan), and an issue resolution (linked inspection from issue). All four cross-module effects fire at once.Each cross-module effect is processed independently and asynchronously. There is no ordering dependency between them. All four effects complete: the action dedup indicator updates, the compliance rule flips to Compliant, the maintenance plan status changes to "Due", and the issue profile shows the linked inspection. Notifications from each module are batched per X-E03.
X-E05A completed inspection is deleted that was linked to: an issue (as evidence), an action (as the source), a compliance rule (as auto-collected evidence), and an asset (as the latest reading).Each linked item handles the deletion gracefully: the issue shows "Linked inspection deleted" (issue remains open), the action retains its data but shows "Source inspection deleted", the compliance rule's evidence is removed (rule may revert to Non-Compliant), and the asset retains the reading in its history but the reading shows "Source inspection deleted". Deletion of inspections linked to other modules requires admin confirmation with a cascade preview.
X-E06Offline sync conflict — a user creates an action offline from an inspection, while another user (online) creates an action from the same issue that the inspection is linked to, targeting the same problem.Both actions are created on sync. The merge feature (Actions module) can consolidate them if they are indeed duplicates.
X-E07A template is used across multiple modules: linked to issue categories, linked to action types, used in compliance rules, scheduled for recurring inspections, and connected to assets. The template is then archived.Before archiving, show a full dependency summary: "This template is referenced by X issue categories, Y action types, Z compliance rules, W schedules, V asset configurations." Archiving proceeds with cascade effects: issue categories show "Template archived" (users pick another), action types disable the "Start inspection" option, compliance rules become "Not Evaluable", schedules are paused, and asset-template links become inactive. Each dependent module's admin is notified.

How deep does the export go?Exports include the primary module's data plus reference IDs and names for linked items from other modules (e.g. action title, issue reference number, asset name, site name). They do not recursively export full data from linked modules. For deep cross-module analysis, use Analytics dashboards or build a custom report combining multiple data sources.
